# Authority-First Research

This skill should gather tutorial inputs from strong sources first and only fall back to secondary sources when the primary layer is incomplete.

## Source Priority

1. Official product or maintainer documentation
2. Standards bodies, regulator publications, or official specifications
3. University, research-lab, or government publications
4. First-party company manuals, changelogs, and API references
5. High-quality secondary explainers only when the primary layer leaves a gap

## Research Rule

- Start from the learning objective, not from a random search query.
- Use `3-7` sources for a first tutorial pass.
- Record the exact URL and the publication or last-updated date when available.
- Explain why each source is trusted.
- Prefer sources that teach the current, supported behavior rather than historical behavior.
- Stop when you can support the tutorial outline with evidence, not when you have exhausted the web.

## What To Capture Per Source

- source title
- source URL
- source type
- publication date or last-updated date
- authority reason
- what the tutorial should borrow
- what the tutorial should not overstate

## Source Types That Usually Count As Authoritative

- official documentation portals
- RFCs and formal standards
- maintainer repositories and release notes
- official help centers
- regulator or compliance guidance
- first-party university or lab documentation

## Source Types That Need Extra Skepticism

- SEO blog posts with no citations
- scraped copies of official docs
- AI-written summaries with no links
- forum answers without maintainer confirmation
- tutorials that are older than the official docs they describe

## Citation Rule

- Cite claims to the source appendix, not to memory.
- Keep quotations short and rare.
- Paraphrase the source in learner-friendly language.
- If the sources disagree, say so and name the tradeoff.
