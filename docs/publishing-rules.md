# yao-open-skills 发布规则

## 1. 发布目标

每次把本地 Skill 收录到 `yao-open-skills` 时，目标不是“把目录搬过去”，而是“产出一个适合公开阅读和复用的版本”。

因此发布流程默认包含筛选、清理、登记和说明页更新四部分。
每个已开源 Skill 还必须补一份面向人类读者的使用说明。

## 2. 可以公开的 Skill

优先收录这类 Skill：

- 通用性强，别人也可能用得上
- 方法完整，不只是临时草稿
- 说明清楚，有明确输入输出
- 不依赖私人账号、专有数据或内部系统
- 你愿意长期维护

## 3. 默认不公开的内容

出现下列情况时，默认不直接公开：

- 包含账号、token、cookie、个人邮箱、私有 API 地址
- 依赖公司内部文档、客户资料、未授权内容
- 目录里混有大量输出结果、抓取内容、缓存、日志
- 仍处于高频实验状态，结构变化很大
- 来源不明，无法确认版权或引用边界

## 4. 导入前检查清单

导入前至少检查：

- `SKILL.md` 是否完整
- `name` 和 `description` 是否清楚
- 是否存在本地运行后生成的输出目录、扫描结果、缓存或 `.command` 操作文件
- 这些输出文件里是否包含绝对路径、私有样本、账号名、令牌或本机环境信息
- 是否有不应公开的脚本、数据、图片、样本
- `scripts/` 是否真的有保留价值
- `references/` 是否包含内部资料
- `assets/` 是否含有无授权字体、商标或模板

默认规则：

- 本地生成的输出物原则上不上传
- 只有明确属于 Skill 源码或必要示例的文件才进入公开仓库

## 5. 导入后的最小动作

一个 Skill 被收录进合集时，至少做这几件事：

1. 复制到 `skills/<slug>/`
2. 清理不该公开的文件
3. 在 `registry/skills.json` 新增或更新登记
4. 为该 Skill 写入 `docs/skills/<slug>.md` 使用说明
5. 运行 README 渲染脚本
6. 明确当前 `sync_status`

## 6. GitHub 同步约定

如果本次任务明确要求同步到 GitHub：

1. 确保本地公开副本已经整理完成
2. 确保 `registry/skills.json` 已记录 `collection_path`
3. 如果已推送公开仓库，写入 `github_repo` 和 `github_url`
4. 写入或更新 `last_synced_at`
5. 若本地源 Skill 后续发生变化，则把 `sync_status` 调整为 `needs-update`

默认远程仓库为：

- `https://github.com/yaojingang/yao-open-skills`

如果本地集合发生增删改，并且这些变更是准备公开的版本，就应在整理完成后提交并推送到这个远程仓库。

## 7. README 说明页更新约定

README 首页至少要反映：

- 这个 Skill 是否已收录
- 使用说明入口
- 当前生命周期状态
- 当前同步状态
- 集合内路径
- 原始来源路径
- GitHub 链接是否已存在

这些信息以登记表为准，不手工逐条改 README。

## 8. 命名约定

- GitHub 仓库名固定为 `yao-open-skills`
- 集合内目录统一使用 kebab-case slug
- 使用说明统一放在 `docs/skills/<slug>.md`
- slug 一旦公开，尽量不改名
- 本地源路径允许变化，但登记表必须更新

详细命名建议见：

- [naming-conventions.md](naming-conventions.md)

## 9. 冲突处理

如果某个本地 Skill 同时满足“值得公开”和“包含敏感材料”，优先做分离：

- 保留公开核心
- 拆掉私有 references
- 删除输出物
- 用公开说明替代私有上下文

做不到干净分离时，就不要发布。
